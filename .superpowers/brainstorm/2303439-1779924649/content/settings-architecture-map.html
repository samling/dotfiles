Registry-First Settings Architecture

Proposed data flow: settings, widgets, popouts, and UI controls all originate from central metadata.

Source of Truth and Generated UI

Settings Registry

Field metadata, defaults, labels, ranges, enum options, validation, grouping.

Widget Registry

Widget IDs, component paths, section constraints, default config, visible settings.

Popout Registry

Surface IDs, lifecycle behavior, IPC actions, one-open-at-a-time rules.

↓

Settings Store

Loads JSON, applies defaults, validates unknown values, persists edits, separates user settings from runtime state.

↓

Settings Window

Generated pages and controls from schemas.

Bar / Widgets

Declarative sections and widget instances.

Popouts / IPC

Coordinator reads registered surfaces and actions.

Why this fits your goal

  • Hardcoded controls become metadata-driven.
  • Widgets and settings can evolve together.
  • Closer to Dank/Noctalia scale without copying their style.

Risk to manage

  • Too much generic UI can feel bland.
  • Registry scope must be limited in the first slice.
  • Migration from current Config must be staged.
